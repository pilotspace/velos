This task addresses two coupled problems. The motorbike jam and the performance bottleneck share the same root: `step_motorbikes_sublane()` processing too many neighbors from an overly wide spatial query.

**A. Fix IDM leader lateral threshold (jam root cause #1):**

In `step_motorbikes_sublane()` (sim.rs ~line 407), the IDM leader detection uses `lateral_dist < 1.5` which is too wide. In dense clusters, a motorbike detects 5+ "leaders" that are actually beside it, causing unnecessary braking cascades.

Change the lateral threshold to `0.8` (roughly one motorbike width + small margin). This means only agents nearly directly ahead are considered IDM leaders:
```rust
// Was: if longitudinal > 0.0 && lateral_dist < 1.5 && longitudinal < idm_gap {
if longitudinal > 0.0 && lateral_dist < 0.8 && longitudinal < idm_gap {
```

**B. Fix intersection dispersal (jam root cause #2):**

The swarming override `if at_red && idm_gap > 2.0` (sim.rs ~line 414) forces a 2.0m gap even for motorbikes that should be accelerating after green. The `at_red` flag is correctly set per signal state, but motorbikes behind the stop line inherit `at_red` from check_signal_red even when the light has just turned green (because offset is still within 15m of intersection).

Add a speed-gate: only apply swarming gap override when the motorbike is actually stopped or nearly stopped:
```rust
// Was: if at_red && idm_gap > 2.0 {
if at_red && speed < 0.5 && idm_gap > 2.0 {
```

This allows motorbikes that are already moving (speed > 0.5 m/s) to use normal IDM gap following even near intersections, enabling dispersal after green.

**C. Reduce spatial query radius (performance + jam):**

In `step_motorbikes_sublane()` (sim.rs ~line 371), reduce the spatial query radius from 10.0m to 6.0m:
```rust
// Was: let nearby = spatial.nearest_within_radius(*agent_pos, 10.0);
let nearby = spatial.nearest_within_radius(*agent_pos, 6.0);
```

Rationale: IDM only needs the closest leader (typically within 3-5m in dense traffic). The sublane model's `MAX_LONGITUDINAL_GAP` is 15m but most relevant neighbors are much closer. 6m captures the critical interaction zone while cutting neighbor count from ~50 to ~15-20 in dense clusters.

**D. Cap neighbor processing count:**

Add a neighbor cap to prevent O(n^2) behavior in dense clusters. After the `nearby` query, take at most 20 neighbors (sorted by distance):

In `spatial.rs`, add a `nearest_within_radius_capped` method:
```rust
/// Find agents within radius, capped at max_count nearest.
pub fn nearest_within_radius_capped(
    &self, pos: [f64; 2], radius: f64, max_count: usize
) -> Vec<&AgentPoint> {
    let radius_sq = radius * radius;
    let mut results: Vec<&AgentPoint> = self.tree
        .locate_within_distance(pos, radius_sq)
        .collect();
    if results.len() > max_count {
        results.sort_by(|a, b| {
            let da = (a.pos[0] - pos[0]).powi(2) + (a.pos[1] - pos[1]).powi(2);
            let db = (b.pos[0] - pos[0]).powi(2) + (b.pos[1] - pos[1]).powi(2);
            da.partial_cmp(&db).unwrap()
        });
        results.truncate(max_count);
    }
    results
}
```

Use this in `step_motorbikes_sublane()`:
```rust
let nearby = spatial.nearest_within_radius_capped(*agent_pos, 6.0, 20);
```

**E. Also reduce sublane MAX_LONGITUDINAL_GAP:**

In `sublane.rs`, reduce `MAX_LONGITUDINAL_GAP` from 15.0 to 10.0. With the reduced spatial radius (6m), neighbors beyond 6m won't be in the query results anyway. The sublane gap-seeking logic only needs nearby neighbors for lateral gap computation:
