**sim_startup.rs changes:**

Add `load_gtfs_bus_stops` function:

```rust
pub(crate) fn load_gtfs_bus_stops(
    road_graph: &RoadGraph,
) -> (Vec<BusStop>, Option<BusSpawner>, HashMap<String, usize>) {
```

Implementation:
1. Determine GTFS path: `std::env::var("VELOS_GTFS_PATH").unwrap_or_else(|_| "data/gtfs".to_string())`
2. Check if path exists and is a directory. If not: `log::info!("No GTFS data found at '{}', bus stops inactive", path)` and return `(Vec::new(), None, HashMap::new())`.
3. Call `velos_demand::load_gtfs_csv(path)`. On error: log warning, return empty.
4. Collect all unique GtfsStops across all routes into a deduped list.
5. Create projection: `EquirectangularProjection::new(10.7756, 106.7019)` -- same reference point as OSM import in app.rs.
6. Call `velos_net::snap_gtfs_stops(&all_stops, road_graph, &proj)` to get `Vec<BusStop>`.
7. Build `stop_id_to_index: HashMap<String, usize>` -- during snapping, track which GtfsStop.stop_id maps to which index in the resulting bus_stops Vec. This requires snap_gtfs_stops to also return this mapping, OR build it by matching names. The cleaner approach: snap_gtfs_stops should return `(Vec<BusStop>, HashMap<String, usize>)`. If Plan 01's snap_gtfs_stops doesn't return this mapping, build it here by iterating the all_stops list and matching against returned BusStops by name (since snap preserves the name).
8. Create `BusSpawner::new(&routes, schedules, &stop_id_to_index)`.
9. Log: `log::info!("GTFS loaded: {} bus stops snapped, {} routes, {} trips", bus_stops.len(), routes.len(), schedules.len())`.
10. Return `(bus_stops, Some(bus_spawner), stop_id_to_index)`.

**sim_lifecycle.rs changes:**

In `spawn_agents(&mut self, dt: f64)`:

1. After the existing OD spawner block (`let requests = self.spawner.generate_spawns(...)` + spawn loop), add GTFS bus spawning:

```rust
// GTFS bus spawning (time-gated by trip departure schedule)
if let Some(ref mut bus_spawner) = self.bus_spawner {
    let bus_requests = bus_spawner.generate_bus_spawns(self.sim_time);
    for bus_req in bus_requests {
        self.spawn_gtfs_bus(&bus_req);
    }
}
```

2. Add `spawn_gtfs_bus(&mut self, req: &BusSpawnRequest)` method:

This method spawns a bus entity using the GTFS route's stop sequence as waypoints. Unlike OD-spawned buses that get random routes, GTFS buses follow a deterministic stop-to-stop path.

Implementation:
- Get the first stop's edge_id from `self.bus_stops[req.stop_indices[0]]` as the starting edge.
- Find source and target nodes for the starting edge via `graph.inner().edge_endpoints(EdgeIndex::new(edge_id as usize))`.
- Build the bus route path: for now, use the first stop's edge as the starting point and find a route from first stop to last stop via CCH or find_route. Actually, per the locked decision "Route-following via stop edges: buses follow their route's stop edge sequence as waypoints", the bus should navigate between consecutive stops. For the initial implementation, compute the shortest path from first stop node to last stop node via `velos_net::find_route()` (CCH is more efficient but find_route is simpler and startup-only cost is acceptable).
- Alternatively, if CCH is available (`self.reroute.cch_router`), use `query_with_path` between the first and last stop nodes for a better path. But for correctness and simplicity, use the simple approach: compute path from origin node (first stop) to destination node (last stop).
- Spawn the bus entity with the same component pattern as the existing bus spawn in `spawn_single_agent` (line 192-208), but with `BusState::new(req.stop_indices.clone())` using the GTFS-derived stop indices instead of the edge-matching heuristic.
- Use `VehicleType::Bus`, `CarFollowingModel::Idm` (70%) or `Krauss` (30%) per existing pattern.
- Log at debug level: `log::debug!("Spawned GTFS bus: trip={}, route={}, stops={}", req.trip_id, req.route_id, req.stop_indices.len())`.

Key difference from existing bus spawning: existing code in `spawn_single_agent` computes bus_stop_indices by matching route edges against ALL bus_stops (lines 102-114). GTFS buses get their stop_indices directly from the BusSpawnRequest (precomputed per-route). This is more accurate -- they visit only their route's stops.
