    - Test: contract_graph on a 4-node diamond graph (A-B, A-C, B-D, C-D) produces exactly 1 shortcut (A-D through B or C)
    - Test: contract_graph preserves all original edges as non-shortcut CCH edges

    - Test: CCHRouter::from_graph on a 10-node grid produces shortcut count < 3x original edge count
    - Test: CCHRouter correctness -- for every pair of connected nodes, the shortest path through CCH upward graph has the same cost as A* on the original graph (test on small graphs)
  </behavior>
  <action>
    Create `crates/velos-net/src/cch/` module directory with four files.

    **ordering.rs:** Implement `compute_ordering(graph: &DiGraph<RoadNode, RoadEdge>) -> Vec<u32>` using nested dissection with BFS-based balanced bisection (reuse the pattern from Phase 5's METIS fallback). Algorithm:
    1. Recursively bisect the graph using BFS from a peripheral node
    2. Separator nodes (edges crossing the bisection) get highest ranks (contracted last)
    3. Recurse into each partition
    4. Base case: partition size <= 10, order by degree ascending
    The output Vec maps node_index -> rank.

    **topology.rs:** Implement `contract_graph(graph: &DiGraph<RoadNode, RoadEdge>, order: &[u32]) -> CCHTopology`. Algorithm:
    1. Process nodes in order (lowest rank first = contracted first)
    2. For each contracted node v, check all pairs of neighbors (u, w) where rank(u) > rank(v) and rank(w) > rank(v)
    3. If no witness path exists through nodes with rank > rank(v), add shortcut (u, w) with middle=v
    4. Build forward_star and backward_star (CSR format) indexed by rank: forward contains upward edges (to higher rank), backward contains upward edges (from lower rank looking up)
    5. Store shortcut_middle: Vec<Option<u32>> for path unpacking

    **mod.rs:** Define `CCHRouter` struct with fields from the research doc:
    ```rust
    pub struct CCHRouter {
        node_order: Vec<u32>,        // node -> rank
        rank_to_node: Vec<u32>,      // rank -> node
        forward_head: Vec<u32>,      // CSR: target nodes of upward edges
        forward_first_out: Vec<u32>, // CSR: index into forward_head per node
        forward_weight: Vec<f32>,    // weights (set during customization, initialized to f32::INFINITY)
        backward_head: Vec<u32>,
        backward_first_out: Vec<u32>,
        backward_weight: Vec<f32>,
        shortcut_middle: Vec<Option<u32>>,
        original_edge_to_cch: Vec<usize>,
        node_count: usize,
    }
    ```
    Implement `CCHRouter::from_graph(graph: &RoadGraph) -> Self` that calls compute_ordering then contract_graph.

    **lib.rs:** Add `pub mod cch;` to velos-net's lib.rs.

    Write tests in `crates/velos-net/tests/cch_tests.rs` covering all behaviors above. Use small hand-crafted graphs (diamond, grid, line) for deterministic testing. Include a correctness test comparing CCH topology against brute-force all-pairs on a small graph.
  </action>
  <verify>
    <automated>cargo test -p velos-net --test cch_tests -- --nocapture</automated>
  </verify>
  <done>
    CCHRouter::from_graph constructs valid ordering and shortcut topology. Shortcut count < 3x original edges. All tests pass.
  </done>
