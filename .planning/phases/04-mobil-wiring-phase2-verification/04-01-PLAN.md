**1. Add LaneChangeState component** to `components.rs`:
```rust
/// Active lane-change state for cars executing a MOBIL-triggered lane change.
/// Attached when lane change starts, removed when drift completes.
pub struct LaneChangeState {
    /// Target lane index (0-based from right).
    pub target_lane: u8,
    /// Time remaining for the drift (seconds). Starts at 2.0, counts down.
    pub time_remaining: f64,
    /// Simulation time when lane change started (for cooldown).
    pub started_at: f64,
}
```

**2. Add MOBIL evaluation to `step_vehicles()`** in `sim.rs`:

After the existing IDM acceleration calculation (line 288), before pushing to `updates`, add MOBIL lane-change evaluation for cars NOT currently in a lane change and NOT near a red signal:

- Build `edge_lane_agents: HashMap<(u32, u8), Vec<(Entity, f64)>>` grouping agents by (edge_index, lane) -- extend the existing `edge_agents` build loop to also track lane
- For each car agent, skip MOBIL eval if: `at_red` is true, agent has `LaneChangeState` component, or a 3-second cooldown hasn't elapsed since last lane change
- For left and right adjacent lanes (if they exist -- check `lane_count` from edge weight):
  - Find the leader and follower in the target lane using the `edge_lane_agents` map
  - Compute `accel_current` = IDM accel with current leader (already computed)
  - Compute `accel_target` = IDM accel with target-lane leader (gap/delta_v from target lane leader)
  - Compute `accel_new_follower` = IDM accel of the closest follower in target lane, with the subject as their new leader
  - Set `accel_old_follower` = 0.0 (simplification: assume old follower unaffected)
  - Call `mobil_decision()` -- if true, spawn `LaneChangeState` and `LateralOffset` on the entity

Add `use velos_vehicle::mobil::{mobil_decision, LaneChangeContext};` and `use velos_vehicle::types::default_mobil_params;` to imports.

**3. Add gradual drift logic** in the same `step_vehicles()` function:

For cars that have a `LaneChangeState`:
- Compute target lateral position: `(target_lane as f64 + 0.5) * 3.5` (lane center)
- Compute current lateral from current lane: `(current_lane as f64 + 0.5) * 3.5`
- Use `LateralOffset` to track intermediate position -- drift linearly over 2 seconds
- Decrement `time_remaining` by `dt`
- When `time_remaining <= 0`: update `RoadPosition.lane` to `target_lane`, remove `LaneChangeState` and `LateralOffset` from entity
- Call `apply_lateral_world_offset()` to render the intermediate position

**4. Add helper function** to `sim_helpers.rs`:

Add `find_leader_in_lane()` and `find_follower_in_lane()` functions that search the `edge_lane_agents` map for the nearest agent ahead/behind in a specific lane on a specific edge. Return (gap, speed) tuple. These are used to populate `LaneChangeContext`.

**Important constraints:**
- Lane changes only on segments, not intersections: the `at_red` check already covers intersection proximity (within 15m of signalized node). Additionally, skip MOBIL if `rp.offset_m < 5.0` (too close to start of edge) or `rp.offset_m > edge_length - 20.0` (approaching intersection).
- Cooldown: store last lane-change completion time -- use `LaneChangeState.started_at + 2.0` as the earliest next evaluation time. Can store this as a simple f64 component or check `sim_time - (started_at + 2.0) > 3.0`.
- Keep `sim.rs` under 700 lines. If it exceeds, extract the MOBIL evaluation logic into a new `sim_mobil.rs` helper module (add `mod sim_mobil;` to `lib.rs`).
