    - Test 2: MOBIL rejects lane change when new follower braking exceeds safe_decel (safety criterion)
    - Test 3: No lane change initiated when agent is near intersection (within 15m of edge end at signalized node)
    - Test 4: Cooldown prevents re-evaluation for 3 seconds after a lane change completes
  </behavior>
  <action>
    **1. Add LaneChangeState component** to `crates/velos-core/src/components.rs`:
    ```rust
    pub struct LaneChangeState {
        pub target_lane: u8,
        pub progress: f64,      // 0.0 to 1.0 over 2 seconds
        pub cooldown: f64,      // seconds remaining before next evaluation
        pub origin_lateral: f64, // lateral offset at start of lane change
        pub target_lateral: f64, // lateral offset at target lane center
    }
    ```

    **2. Modify `step_vehicles()` in `sim.rs`** to add MOBIL evaluation:
    - Import `mobil_decision`, `MobilParams`, `LaneChangeContext` from `velos_vehicle::mobil` and `default_mobil_params` from `velos_vehicle::types`
    - Import `LaneChangeState` from `velos_core::components`
    - In the agent query, also fetch `Option<&LaneChangeState>` for each car
    - Group `edge_agents` by `(edge_index, lane)` instead of just `edge_index`, so we can find leaders/followers per lane
    - For cars WITHOUT `LaneChangeState` and with cooldown expired:
      - Skip MOBIL evaluation if near intersection (within 15m of edge end at a signalized node, reuse `check_signal_red` proximity logic)
      - Get lane count from road edge. Skip if single-lane road.
      - For each adjacent lane (lane-1 if lane>0, lane+1 if lane<lane_count-1):
        - Find leader in target lane: nearest agent ahead in `(edge_index, target_lane)` group
        - Find new follower in target lane: nearest agent behind in `(edge_index, target_lane)` group
        - Compute IDM accelerations for `LaneChangeContext` fields using `idm_acceleration()`
        - Call `mobil_decision()`. If true, create `LaneChangeState` with target_lane, progress=0.0, cooldown=0.0
        - Set origin_lateral = current lane center (lane * 3.5 + 1.75), target_lateral = target lane center
        - Attach `LateralOffset` component with lateral_offset = origin_lateral, desired_lateral = target_lateral
    - For cars WITH `LaneChangeState`:
      - Advance `progress += dt / 2.0` (2-second transition)
      - Interpolate lateral_offset = lerp(origin_lateral, target_lateral, progress) -- linear drift per user decision
      - Update `LateralOffset` component
      - If progress >= 1.0: update `RoadPosition.lane` to `target_lane`, remove `LaneChangeState`, remove `LateralOffset`, set cooldown on a new tracking mechanism
    - For cooldown tracking: add a simple `LaneChangeCooldown(f64)` component. Decrement each tick. Remove when <= 0. Skip MOBIL eval if present.

    **3. Add `find_leader_in_lane()` and `find_follower_in_lane()` helpers** to `sim_helpers.rs`:
    - `find_leader_in_lane(entity, rp, lane, edge_lane_agents, speed_map, own_speed) -> (gap, delta_v)`
    - `find_follower_in_lane(entity, rp, lane, edge_lane_agents, speed_map, own_speed) -> (gap, delta_v)`
    - These are variants of `find_leader_static` that filter by specific lane

    **4. Update `sim_lifecycle.rs`** spawning:
    - Cars should NOT get `LateralOffset` at spawn (only during lane change)
    - Cars should NOT get `LaneChangeState` at spawn
    - Ensure car spawn sets `RoadPosition.lane = 0` (or random lane if multi-lane)

    **5. Apply lateral world offset for lane-changing cars** in the update section of `step_vehicles()`:
    - After `apply_vehicle_update()`, if car has `LateralOffset`, call `apply_lateral_world_offset()` (same as motorbikes)

    **6. Write integration test** `crates/velos-vehicle/tests/mobil_integration_tests.rs`:
    - Test MOBIL decision + lane change state machine (unit-level, no ECS needed)
    - Test that `LaneChangeContext` populated from IDM evaluations produces correct decisions
    - Test cooldown prevents rapid oscillation

    NOTE: The `edge_agents` grouping change from `HashMap<u32, Vec<(Entity, f64)>>` to `HashMap<(u32, u8), Vec<(Entity, f64)>>` is the key structural change. Update `find_leader_static` to also work with the new grouping (pass lane parameter) or keep backward compat by also maintaining the per-edge-only grouping for IDM leader finding (current-lane leader).
